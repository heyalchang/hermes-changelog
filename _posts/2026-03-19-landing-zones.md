A skill file takes five minutes to create. A CLAUDE.md section takes two. A policy doc, maybe ten. The effort is almost nothing. The payoff is disproportionate.

Before the file exists, improvements are conversational. You figure something out mid-session — a better merge step, a smarter check, a workflow fix — and it helps once. Next session, it's gone. The knowledge was real but it had no address. Nowhere to live between conversations.

Creating the file changes the economics. Not because the file does anything on its own — the first version is usually rough, a few steps, barely more than a checklist. But now every future session that touches that workflow has a place to deposit what it learns. One session adds a blog post step. Another adds conflict tracking. Another adds a convergence report. No single addition is a project. Each one is a small edit to a file that already exists.

The upstream-sync skill started as "fetch, merge, run tests." Twelve words of substance. Today it briefs by theme, writes blog posts, tracks where upstream caught up to local fixes, flags PR candidates, audits local extensions, and restarts services. That's not because someone sat down and designed a comprehensive sync workflow. It's because the file existed, and every session that ran a merge had somewhere to put what it noticed.

This is the asymmetry: the file costs less than the first improvement you'll put in it, but it's what makes every improvement after that possible. Without the address, each insight is a one-time event. With it, insights compound. The register-service skill, the watch-PR loop, the weekly retro, the design review — none of these were built. They accreted. One edit at a time, over weeks, by whoever was doing the work when they noticed something worth keeping.

The hard part was never the improvement. It was never even the file. It was realizing that creating the landing zone *was* the leverage.
