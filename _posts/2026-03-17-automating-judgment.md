Skills automate procedures. Policies automate judgment. Together they're what makes agents actually autonomous.

I noticed this today while cross-pollinating between two projects. One had solved a session continuity problem — how to not lose context when a conversation dies mid-stream. The other project needed the same thing but lighter. The agent read the first project's solution, understood the principle behind it, and built something different. Not a copy. A different design for a different context.

It wasn't following a template. It was reading a policy — a short document that said *why* session state matters and *when* to write it, without prescribing *how*. The agent made its own design choice. That's the distinction. A skill is a procedure: do these steps, in this order, check these things. A policy is a principle: here's what matters and why. The agent figures out what that looks like for the project it's in.

I keep solving the same kinds of problems differently across projects because the contexts are different. The right answer for a multi-session feature branch isn't the right answer for a single-session daemon fix. What's reusable isn't the solution — it's the judgment that chose it. The policy captures that judgment. The agent applies it.

This is what I mean by automating myself in pieces. Not the work — I've been automating work for a while. The part of me that notices a class of problem, decides on an approach, and adapts it to the situation. That's the expensive part. That's what doesn't scale when I'm the only one who carries it. A policy directory with ten good documents and an agent that reads them before starting — that's a version of me that's always in the room.
